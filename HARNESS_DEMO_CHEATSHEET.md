# Шпаргалка на один экран

> Держать открытым на втором мониторе во время демо.

## Ключевые числа — заучить наизусть

| Метрика | Значение | Контекст |
|---------|----------|----------|
| Recall@20 в training | **100%** | Модель "якобы" идеальна |
| Real early capture | **6.7%** | Что бот реально делает |
| 7-day baseline | **41.9%** | Регрессия −35pp |
| Training-to-live gap | **+66.7%** | Главный сигнал |
| False positives | **13/18 = 72%** | 3/4 сигналов мимо |
| Стресс-тест pass | **3/3** | После фикса L6 бага |
| LOC всего | **~3300** | 11 файлов |
| Daily цикл | **6 секунд** | Полный orchestrator |

## Ключевые фразы — повторить 3 раза

1. *"Главный вызов оказался не в коде бота, а в том,
   как заставить агентов честно его улучшать."*

2. *"Один агент с полным контекстом неизбежно rationalize свои решения —
   узкий контекст у каждого слоя это и есть защита от confirmation bias."*

3. *"Скрипт не равен pipeline. Pipeline — это дисциплина:
   каждое решение фиксируется, каждое предсказание проверяется,
   каждый промах кладётся в память."*

4. *"Стресс-тест поймал реальный баг в первый же прогон.
   Без него я бы залил его в прод."*

## Anti-patterns которые я обнаружил

- ❌ Тренировочные метрики как KPI ("recall@20 100%" пока бот теряет деньги)
- ❌ Один агент валидирует свои же предложения
- ❌ Изменения без явного критерия отката
- ❌ Память хранится в голове разработчика, не в файлах
- ❌ "Бэктест прошёл" без hold-out test set

## Что они скорее всего спросят

**Q: Почему не один большой prompt с chain-of-thought?**
A: Любой агент со всем контекстом rationalize выводы. Узкий контекст
у каждого слоя — единственный способ режа bias. Это не дороже —
это другой trade-off.

**Q: А если L5 ошибётся?**
A: После него человек + L7 через 7 дней проверит outcome.
Многослойность означает, что одна ошибка не пробивает систему.

**Q: Сколько времени делал?**
A: Один день архитектура + один день реализация всех 7 слоёв.
Большая часть работы — продумать контракты между слоями, а не код.

**Q: Что в этом нового по сравнению со стандартным MLOps?**
A: MLOps валидирует модели. Мой pipeline валидирует **решения агентов**
(включая решения о том, что менять в системе). Это уровнем выше.

**Q: Можно ли это масштабировать на не-крипту?**
A: Архитектура агностична. Любая система, где training_metric ≠
deployment_metric и есть возможность backtest. Например — RAG-системы,
рекомендательные движки, чат-боты с тюнингом промптов.

**Q: Что осталось ручным?**
A: Бэктест-валидаторы пока написаны для одного типа диффа.
Для нового — один день работы. Архитектура для расширения готова.

## Чего НЕ делать в демо

- ❌ Углубляться в специфику crypto (BUY/SELL, ranker_proba, ATR)
- ❌ Извиняться за то, что часть работает на ручке (это нормально)
- ❌ Читать слайды дословно — рассказывать
- ❌ Перегружать терминами — "north star metric" вместо "early_capture"

## Чего ДЕЛАТЬ

- ✅ Делать паузы после каждой цифры
- ✅ Указывать на gap между Training и Deployment как на главный инсайт
- ✅ Подчёркивать, что pipeline уже поймал реальный баг (slide 8)
- ✅ Рассказать про hit_rate < 60% как самоконтроль pipeline
- ✅ Закончить с примером применения вне крипты
