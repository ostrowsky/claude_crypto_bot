# Harness для непрерывного улучшения AI-агента

**Демо для BOS.PRO · 7–10 минут · 10 слайдов**

> Замечание для себя: каждый слайд ≈ 45–60 сек. Делать паузу после каждой
> цифры. Не углубляться в крипту — её упомянуть как контекст и сразу
> переходить к мета-задаче (улучшение работы агента).

## Slide 3 · Что значит "честная метрика" (1 мин)

**Заголовок:** Я разделил "здоровье обучения" и "здоровье работы"

**Раньше — одна метрика:** "точность модели" — она была отличная и
маскировала всё остальное.

**Стало — северная звезда + дельта между двумя уровнями:**

```
Training Health  (хорошо ли обучилась модель?)
        │
        │ ← gap здесь = главная диагностика
        ▼
Deployment Health (то ли модель делает в реальной работе?)
        │
        ▼
Exit Quality      (нормально ли закрываем позиции?)
```

**Ключевой инсайт:** разрыв между уровнями ≠ ошибка модели. Это
проблема downstream — фильтры, скоринг, принятие решений ломают то,
что модель правильно нашла. Без явного измерения gap'а — невидимо.

## Slide 4 · 7-слойный pipeline (1.5 мин)

**Заголовок:** Каждый слой — отдельный агент с узким контекстом

```
┌─────────────────────────────────────────────────┐
│ L1  Сбор метрик (ежедневно, 03:00)              │  → JSON + Markdown
│     Видит: 5 источников данных                  │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│ L2  Генератор гипотез (еженедельно)             │  → 1-3 предложения
│     Видит: 7 дней отчётов + память неудач       │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│ L3  Бэктест-валидатор                           │  → accept / reject
│     Видит: 60 дней истории                      │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│ L4  Shadow-режим (7 дней рядом с продом)        │  → diff отчёт
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│ L5  Слепой ревьюер                              │  → independent verdict
│     Видит ТОЛЬКО: числа до/после                │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│ L6  Человек (я)                                 │  → flip flag
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│ L7  Авто-rollback через 7 дней если не работает │
└─────────────────────────────────────────────────┘
```

**Принцип:** разный контекст у каждого слоя → агенты не сговариваются.

## Slide 5 · L1+L2: От метрик до гипотез (1 мин)

**Заголовок:** Объективные данные → проранжированные предложения

**L1 — отчёт здоровья:**
- Cливает 5 источников в один JSON + Markdown
- Светофор по каждой метрике (🟢🟡🔴🚨)
- 7-дневный baseline для каждого числа
- "Red flags" со ссылкой на конкретные кейсы

**Пример из реального запуска сегодня:**
```
🚨 early_capture: 6.7% (7-day baseline: 41.9%, регрессия -35pp)
🚨 training_to_live gap: +66.7%  ← главный диагностический сигнал
🚨 5 over-blocking filters detected
```

**L2 — генератор гипотез:**
- Читает только устойчивые red flags (≥4 дня из 7)
- Память "что уже пробовали" блокирует повторы за 30 дней
- Список "не трогать" защищает работающие компоненты
- Выдаёт ≤3 предложения с критерием отката

## Slide 6 · L3+L4: Проверка гипотез без риска (1.5 мин)

**Заголовок:** Бэктест на истории + shadow рядом с продом

**L3 — Backtest-валидатор:**
- Берёт диф из гипотезы, делает Pareto-sweep вокруг значения
- 5+ точек на 60-дневной истории
- Чёткие критерии отказа: если ключевая метрика падает >2pp → reject

**Реальный пример сегодня:**
Гипотеза: понизить порог фильтра с 56 до 28
L3 прогнал 840 заблокированных событий за 60 дней:
- При новом пороге 727 событий разблокировались
- Но **96% из них** имели низкое качество по второй метрике
- **Verdict: reject** (риск ложных срабатываний выше +5pp)

Без L3 я бы это применил и узнал через неделю.

**L4 — Shadow-режим:**
- Изменение работает рядом с продом, но не влияет на реальные решения
- Бот логирует "что бы я сделал с изменением" vs "что сделал сейчас"
- 7 дней accept gate: ≥N событий + метрика shadow ≥ метрика прод

## Slide 7 · L5+L6+L7: Слепой ревью, человек, авто-откат (1.5 мин)

**Заголовок:** Защита от confirmation bias на финише

**L5 — Слепой ревьюер:**
- Получает ТОЛЬКО: ключ изменения, дату, ожидаемую дельту
- НЕ получает: hypothesis_id, обоснование, отчёты L3/L4
- Видит только числа за 7 дней до и 7 дней после
- Самостоятельно решает: подтвердилась ли гипотеза + нет ли регрессий в других метриках

*Зачем именно слепой:* если ревьюер видит обоснование, он повторяет
логику предыдущих слоёв. Слепой режет confirmation bias.

**L6 — Человек:**
- Финальная кнопка y/n
- Safety-checks: locked configs, do_not_touch list, статус валидаций
- Запись в decisions.jsonl + already_tried.jsonl (память для L2)
- Готовый diff для применения + commit message

**L7 — Авто-rollback:**
- Каждый день читает decisions.jsonl
- Для решений старше 7 дней: проверяет реальную дельту vs ожидаемую
- Если не сбылось → рекомендация отката + запись в память
- Считает мета-метрику pipeline: `hit_rate = hits / (hits + misses)`
- Если hit_rate < 60% за квартал — pipeline хуже монетки, требует пересмотра

## Slide 8 · Самопроверка: ловит ли pipeline свои же баги? (1 мин)

**Заголовок:** Стресс-тест с синтетической регрессией

Раз в месяц pipeline сам подсовывает себе **заведомо плохие** гипотезы
и проверяет, что они отклоняются:

1. Радикальное ослабление порога — должно дать reject от L3
2. Попытка изменить locked config — должна блокироваться L6
3. Попытка ослабить gate из "не трогать" — должна блокироваться L3

**Первый прогон сегодня:** 2/3 pass — **тест поймал реальный баг**:
флаг `--approve` обходил safety-checks (был алиасом для `--force`).

Исправил → второй прогон → 3/3 pass.

*Это лучшая иллюстрация смысла стресс-теста: без него я бы залил
этот баг в прод и не узнал.*

## Slide 9 · Главные инсайты (1 мин)

**Заголовок:** Что я понял про harnessing агентов

1. **Один агент = ноль гарантий.** Любой агент с полным контекстом будет
   подгонять выводы. Только разделение ролей с **узким контекстом**
   у каждого даёт честность.

2. **Скрипт ≠ pipeline.** Pipeline — это **дисциплина**: каждое
   решение фиксируется, каждое предсказание проверяется, каждый
   промах кладётся в память. Без этого — красивая схема без эффекта.

3. **Тренировочные метрики врут.** Не специально — но они оптимизируются
   на одно, а пользователь оплачивает другое. **Gap между training и
   deployment** — единственная объективная диагностика.

4. **Стресс-тест на синтетике обязателен.** Иначе теоретическая
   защита от регрессии = 0. Pipeline должен ловить свои же баги.

5. **Память важнее интеллекта.** Без `already_tried` агенты ходят по
   кругу. Без `do_not_touch` — ломают то, что работает.

Это закрывает loop honestly.

## Резерв на Q&A — частые вопросы

**Q:** Почему именно multi-agent, а не один большой prompt?
**A:** Один агент со всем контекстом неизбежно rationalize свои решения.
Каждый слой ослабляет confirmation bias. Это не дороже — это просто
другой trade-off: больше latency, меньше bias.

**Q:** Что если L5 (слепой) ошибётся?
**A:** Он не последний фильтр. После него человек. Кроме того L7 через
7 дней проверит actual outcome — если L5 ошибся, L7 поднимет rollback.

**Q:** Как масштабировать на не-крипту?
**A:** Архитектура агностична: любая система, где есть
training_metric ≠ deployment_metric и где можно делать backtest.
Например — рекомендательные системы, чат-боты с тюнингом промптов,
ML-сервисы с A/B-роллаутом.

**Q:** Что пока работает на ручке?
**A:** Бэктест-валидаторы пока написаны для одного типа диффа
(numerical config). Для нового типа диффа нужен ~один день, чтобы
написать replay-скрипт. Архитектура для этого готова: один файл, одна
функция, регистрация в dispatcher dict.

**Q:** Хочешь ли ты применить это к работе агентов вне крипты?
**A:** Это и есть мотивация откликнуться к вам — у меня есть
готовый паттерн, но крипто-домен сужает применение. Хочу прокачать на
сложных доменах + научиться у людей, которые делают это профессионально.
